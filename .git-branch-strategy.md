# Git Branch Strategy

이 프로젝트는 Git Flow 전략을 사용합니다.

## 브랜치 구조

### 주요 브랜치
- **main**: 프로덕션 배포용 (안정화된 코드만)
- **develop**: 개발용 메인 브랜치 (자동 빌드 실행)

### 보조 브랜치
- **feature/***: 새로운 기능 개발
- **release/***: 릴리스 준비 및 버그 수정
- **hotfix/***: 프로덕션 긴급 수정

## 브랜치 사용 규칙

### Feature 브랜치
```bash
# 기능 개발 시작
git checkout develop
git pull origin develop
git checkout -b feature/my-new-feature

# 작업 후
git add .
git commit -m "feat: 새로운 기능 추가"
git push origin feature/my-new-feature

# Pull Request 생성 (feature → develop)
```

### Develop 브랜치
- 모든 기능 개발의 통합 브랜치
- 이 브랜치에 푸시되면 자동으로 빌드 실행
- 빌드 아티팩트는 GitHub Actions에서 다운로드 가능

### Release 브랜치
```bash
# 릴리스 준비
git checkout develop
git checkout -b release/v2.1.0

# 버그 수정 및 테스트
git add .
git commit -m "fix: 릴리스 버그 수정"

# develop과 main에 병합
git checkout develop
git merge release/v2.1.0
git push origin develop

git checkout main
git merge release/v2.1.0
git tag v2.1.0
git push origin main --tags
```

### Main 브랜치
- 프로덕션 배포용
- 태그가 푸시되면 자동으로 릴리스 생성
- 안정화된 코드만 포함

### Hotfix 브랜치
```bash
# 프로덕션 긴급 수정
git checkout main
git checkout -b hotfix/critical-bug-fix

# 수정 후
git checkout main
git merge hotfix/critical-bug-fix
git tag v2.0.1
git push origin main --tags

# develop에도 병합
git checkout develop
git merge hotfix/critical-bug-fix
git push origin develop
```

## 워크플로우

1. **기능 개발**: `feature/feature-name` 브랜치에서 작업
2. **PR 생성**: `feature/feature-name` → `develop`
3. **리뷰 및 병합**: PR 승인 후 `develop`에 병합
4. **자동 빌드**: `develop`에 푸시되면 자동 빌드 실행
5. **릴리스 준비**: `release/v2.x.x` 브랜치 생성
6. **프로덕션 배포**: `main`에 태그 푸시 시 자동 릴리스

## 브랜치 보호 규칙 (권장)

GitHub 저장소 설정에서 브랜치 보호 규칙을 설정하세요:

### Main 브랜치
- PR 필수
- 최소 1명의 승인 필요
- CI 통과 필수

### Develop 브랜치
- PR 권장 (선택)
- CI 통과 필수
